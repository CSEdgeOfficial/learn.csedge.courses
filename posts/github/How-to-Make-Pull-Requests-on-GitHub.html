How to Make Pull Requests on GitHub

Pull requests are a fundamental aspect of collaborative development on GitHub. They allow you to propose changes to a repository and discuss them with your team before merging them into the main branch. In this article, we'll walk through the process of creating a pull request using the CSEdgeOfficial/Web-Development-Internship repository as an example.

1. Fork the Repository

Visit the CSEdgeOfficial/Web-Development-Internship repository on GitHub. In the top-right corner, click on the "Fork" button. This will create a copy of the repository under your GitHub account. It's a good practice to fork a repository when you want to contribute changes, as it keeps the original repository clean and allows for easier collaboration.

2. Create a Branch

Once your fork is ready, create a new branch to work on. Branches allow you to work on changes in isolation, ensuring that your main branch remains stable. Click on the "Branches" tab, then click on the "New branch" button. Give your branch a descriptive name, preferably related to the changes you'll be making.

3. Make Your Changes

With your new branch selected, you can now make your desired changes. Edit files, add new features, fix bugs, or make any other contributions you wish. As you make changes, consider committing them in small, logical chunks. This makes it easier to review and understand your pull request later on.

Pro Tip:

Use descriptive commit messages! A good commit message explains what changes were made and why. This helps reviewers understand your thought process and makes it easier to accept your pull request.

4. Open a Pull Request

When you're happy with your changes and ready to propose them for review, navigate to the "Pull requests" tab and click on the "New pull request" button. GitHub will compare your branch to the original repository's main branch and show you the differences. Review the changes and make sure everything looks correct.

5. Write a Clear Description

A good pull request description helps reviewers understand the purpose and scope of your changes. Explain what you've changed, why you've made those changes, and any relevant details that will aid the review process. If your pull request addresses specific issues or feature requests, be sure to reference them using keywords like "Fixes #123" or "Closes #456."

6. Request Reviewers

GitHub allows you to request specific people to review your pull request. If you're working with a team, you can @mention them in the description or in a comment to notify them. You can also request reviews from the "Reviewers" tab of your pull request. Reviews are crucial for ensuring code quality and maintaining a healthy codebase.

7. Discuss and Iterate

The review process is a collaborative effort. Your reviewers may have suggestions, questions, or requests for changes. Discuss these points and make the necessary adjustments to your branch. You can push new commits to your branch, and they will automatically be added to the pull request. This iterative process ensures that the final merged code is of high quality.

8. Merge Your Changes

Once your pull request has been approved and all discussions are resolved, it's time to merge your changes into the main branch. If you have the necessary permissions, you can merge the pull request yourself. Otherwise, a project maintainer or team member with the required permissions will do it for you. Congratulations, your changes are now part of the repository!

9. Clean Up

After your pull request is merged, it's a good idea to clean up your branches. You can delete the branch you created for your changes, both on your fork and the original repository. This keeps your repository tidy and prevents confusion in the future.

Pull requests are a powerful tool for collaborative development, and they form the backbone of open-source projects on GitHub. By following these steps and contributing to repositories like CSEdgeOfficial/Web-Development-Internship, you're not just learning but also actively participating in the open-source community.
